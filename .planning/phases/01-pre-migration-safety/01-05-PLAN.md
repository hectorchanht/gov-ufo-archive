<objective>
Execute PMS-02 — replace the current versioned-shell SW at `/sw.js` on the live GH Pages origin with a kill-switch SW that (a) unregisters itself, (b) deletes every cache from the old SW, and (c) signals all open clients to soft-reload. Per `.planning/research/PITFALLS.md` Pitfall #1, without this kill-switch the Phase 6 GH-Pages → CF-Pages cutover serves returning users a stale shell whose old SW intercepts navigations against the new origin — producing the "site is empty / PDFs say 404" failure mode the project must NEVER ship.

Pitfall #1's full mitigation has TWO parts that both ship in this plan:
1. A kill-switch `sw.js` (Task 1) that nukes the old caches + unregisters itself.
2. `updateViaCache: 'none'` on every `navigator.serviceWorker.register('/sw.js', ...)` call across the HTML tree (Task 2 — added per revision blocker). Without (2), the browser HTTP-caches `sw.js` and returning users keep getting the OLD (versioned-shell) SW for hours-to-days after the kill-switch ships, defeating the 14-day Phase 6 countdown the kill-switch is supposed to bound.

Per `.planning/phases/01-pre-migration-safety/01-CONTEXT.md` D-05..D-08 (with D-18 confirming this entire commit lands on `main`, NOT on any `ssg-migration` branch — that branch is created in Phase 2):
- **D-05**: Behaviour is **full nuke** — unregister + delete every cache + postMessage soft-reload + skipWaiting + claim.
- **D-06**: Coexist strategy is **replace in-place** at `/sw.js`. Same scope, same URL. Browser auto-installs over the old SW on next visit.
- **D-07**: No self-disable timer. Kill-switch stays installed at the old origin until Phase 6 cutover, at which point the new (real) SW with a DIFFERENT cache-name prefix takes over the same `/sw.js` URL.
- **D-08**: The 14-day gate measures from kill-switch DEPLOY date on GH Pages (= push-to-`main` date), not from per-user install date. Phase 6 cutover may not occur earlier than (deploy date + 14 days).

Output: A rewritten `sw.js` that IS the kill-switch + a minimally-adapted `scripts/build-sw.py` that stamps the kill-switch's version constant + a `scripts/patch-sw-registration.py` patcher that has rewritten every tracked HTML file's SW registration to include `updateViaCache: 'none'` + a verified DevTools observation that the kill-switch is doing the right thing on a returning-user profile.

<!-- The kill-switch SW contract being installed (per D-05): -->

```javascript
// VERSION stamping is preserved — build-sw.py rewrites this line on every deploy.
// Cache prefix is 'realufo-killswitch-' so it cannot collide with the future Phase 6 SW.
const VERSION = '<git-short-sha>-<YYYYMMDD>';

// install: skipWaiting so the kill-switch takes over immediately
self.addEventListener('install', (event) => {
  self.skipWaiting();
});

// activate: full nuke
self.addEventListener('activate', (event) => {
  event.waitUntil((async () => {
    // (1) unregister self
    await self.registration.unregister();
    // (2) delete every cache (the old shell + data + img caches, plus any orphans)
    const keys = await caches.keys();
    await Promise.all(keys.map((k) => caches.delete(k)));
    // (3) signal all controlled clients to soft-reload
    const clientsList = await self.clients.matchAll({ type: 'window' });
    for (const client of clientsList) {
      client.postMessage({ type: 'sw-killswitch-reload' });
    }
    // (4) claim so future page loads see the no-controller state
    await self.clients.claim();
  })());
});

// fetch: no handler (or a pass-through). Per D-05 the goal is for the SW to
// stop intercepting requests entirely after one reload.

// No message handler is required; the kill-switch is one-shot.
```

    ```python
    #!/usr/bin/env python3
    """
    Idempotent patcher: rewrites every `navigator.serviceWorker.register("/sw.js")`
    call across `git ls-files '*.html'` to include `{ updateViaCache: 'none' }`.

    Per .planning/research/PITFALLS.md Pitfall #1 mitigation #2 — without this option
    the browser HTTP-caches sw.js, defeating the Phase 1 kill-switch.

    Safe to re-run: files already carrying the option are left untouched.
    """
    import subprocess
    import sys
    from pathlib import Path

    BEFORE = 'navigator.serviceWorker.register("/sw.js")'
    AFTER  = 'navigator.serviceWorker.register("/sw.js", { updateViaCache: \'none\' })'
    GUARD  = "updateViaCache"  # presence of this substring means already-patched

    def main() -> int:
        # Use git to enumerate tracked HTML files — matches the project convention
        # for build scripts (per CLAUDE.md §6.2: "git ls-files" not os.listdir).
        result = subprocess.run(
            ["git", "ls-files", "*.html"],
            capture_output=True, text=True, check=True
        )
        files = [Path(p) for p in result.stdout.split() if p]

        patched, skipped_already_ok, skipped_no_register = 0, 0, 0
        for path in files:
            text = path.read_text(encoding="utf-8")
            if BEFORE not in text:
                # File either has no SW registration OR already carries the option.
                if GUARD in text and AFTER in text:
                    skipped_already_ok += 1
                else:
                    skipped_no_register += 1
                continue
            new_text = text.replace(BEFORE, AFTER)
            if new_text == text:  # paranoia: substitution did nothing
                skipped_no_register += 1
                continue
            path.write_text(new_text, encoding="utf-8")
            patched += 1

        print(f"patched={patched} already_ok={skipped_already_ok} no_register={skipped_no_register} total={len(files)}")
        return 0

    if __name__ == "__main__":
        sys.exit(main())
    ```
