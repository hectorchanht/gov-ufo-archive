<objective>
Execute PMS-04 — drop the DNS TTL on `realufo.org` to 300 s and start the 7-day countdown gating Phase 6 cutover. Per `.planning/research/PITFALLS.md` Pitfall #15, dropping TTL same-day-as-cutover means resolvers continue serving the old (high-TTL) record for hours, producing a multi-hour split-brain where half of users hit GH Pages with a stale SW (Pitfall #1) and half hit Cloudflare Pages with the new one. The mitigation is to drop TTL ≥ 7 days BEFORE cutover — and Phase 6 cannot legitimately begin until that gate is satisfied.

Per `.planning/phases/01-pre-migration-safety/01-CONTEXT.md` D-17, the DNS provider is not yet confirmed in this repo. Task 1 discovers it (likely Cloudflare DNS since the user has chosen Cloudflare Pages for Phase 2+, but it could be the domain registrar or a third-party DNS host); Task 2 performs the TTL change and verifies via `dig`.

Output: A decision document at `.planning/decisions/dns-ttl.md` that records WHICH DNS provider serves realufo.org, WHEN the TTL was dropped to 300 s, and the `dig +noall +answer realufo.org` outputs confirming the change has propagated. Phase 6's cutover-gate check reads the TTL-change timestamp from this file.

This is the SECOND ADR in the `.planning/decisions/` folder (Akamai spike is the first, in plan 03). Two ADRs from Phase 1 establishes the convention.

<!-- DNS discovery commands -->

The current `realufo.org` apex points at GitHub Pages. To identify the authoritative DNS provider:

```
dig +trace realufo.org           # shows the full delegation chain — the NS records at the bottom are the authoritative nameservers
dig NS realufo.org +short         # short form: just the nameserver names
whois realufo.org                 # registrar may also be the DNS host (common with Cloudflare Registrar)
```

Authoritative-nameserver patterns to recognise:
- `*.ns.cloudflare.com` → Cloudflare DNS (most likely given Phase 2 chose Cloudflare Pages)
- `dns*.registrar-servers.com` → Namecheap
- `ns*.googledomains.com` → Google Domains
- `pdns*.ultradns.net` → UltraDNS (enterprise; unlikely)
- `*.awsdns-*` → AWS Route 53

The exact provider determines the TTL-change UI (each has a different dashboard); the README in this plan accommodates whichever surfaces.

Current TTL — verify before the change:
```
dig +noall +answer realufo.org
;; e.g.: realufo.org.  3600  IN  A  185.199.108.153   # the 3600 is the TTL in seconds
```

After the change, the same command should show `300` (or a value descending from the old TTL toward 300; the old high-TTL records may continue to be served by caching resolvers until they expire).
