<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Operator browser ↔ Cloudflare dashboard | Operator authenticates via CF SSO; CF dashboard is the only path to project creation (no public API for Pages project bootstrap without an account-scoped API token). |
| GitHub repo ↔ CF Pages GitHub App | CF installs a first-party GitHub App into the repo at project-creation time; subsequent webhook traffic is GH → CF only, signed by GitHub. |
| Public internet ↔ *.realufo.pages.dev | Static origin serving committed repo content; identical trust surface to today's GitHub Pages — no new attack surface introduced by this plan. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | `_headers` rules pushed without review | mitigate | `_headers` lives in repo; PR review is the gate. Cloudflare reads it on every deploy — no out-of-band edit possible without a commit. |
| T-02-02 | Information disclosure | HSTS without preload + no CSP | accept | D-07/D-08 explicitly defer CSP + HSTS preload to Phase 6 cutover. Site is read-only public archive — no authenticated surface to protect this phase. |
| T-02-03 | Denial of service | CF Pages free-tier build limit (500/month) | accept | research/STACK.md notes ~30 builds/month projected. Plenty of headroom; revisit only if Phase 4 introduces per-commit preview deploys at unexpected scale. |
| T-02-04 | Elevation of privilege | CF Pages GitHub App install scope | mitigate | Operator authorizes App against repo `hectorchanht/gov-ufo-archive` only — not org-wide. Documented in Task 1 step 3 + `.planning/decisions/cf-pages-project.md`. |
| T-02-05 | Spoofing | Stale `/sw.js` from CF Pages cache poisoning a future SW update | mitigate | `_headers` pins `/sw.js` to `Cache-Control: no-cache, no-store, must-revalidate` (D-08), defending against research/PITFALLS.md Pitfall #1. |
</threat_model>
